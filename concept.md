# Motiva — コンセプトドキュメント

> なぜ Motiva が動機構造を持つのか、そのメカニズムはどう機能するか。
> 背景となる「なぜ/何を」は `vision.md` を、実装の詳細は `design.md` を参照。

---

## 1. コアメカニズム: 4要素モデル

Motiva の中心には、エージェントの「いま・ここ・次」を定義する 4 要素がある。

```
【メタ動機レイヤー】
  好奇心（ゴールを生成・更新）← 学習フィードバック（完了結果の蓄積）
        │
        ▼
┌─────────────────────────────────────┐
│  ゴール（達成閾値付き）              │
│    ↕                                │
│  現在地（観測ベース + 確信度）       │
│    ↕                                │
│  ギャップ（差分 = 次の動機）         │
└─────────────────────────────────────┘
        │
       制約（3軸）
        ├─ 発散防止（量の制御）
        ├─ 不確実性制約（質の制御）
        └─ リソース制約
        │
        ▼
      タスク
```

### 要素 1: ゴール

ゴールは「こうあるべき状態」を表す宣言であり、Motiva が永続的に保持する。
重要なのは、ゴールは単なる文字列ではなく**達成閾値**を伴う点だ。「何がどのレベルに達したら完了か」が明示されていなければ、エージェントは際限なく作業を続けるか、あるいは何もしていないうちに完了と誤判断する。

ゴールの評価は複数次元で行われる。たとえば「TODO アプリを作る」であれば、`feature_completeness`（機能の網羅度）・`test_coverage`（テストカバレッジ）・`docs_coverage`（ドキュメント整備度）のような次元それぞれに閾値を設定する。単一の「進捗率」だけでは不十分であることは、後述の「進捗観測」セクションで詳しく説明する。

### 要素 2: 現在地（状態ベクトル + 確信度）

現在地は**自己申告ではなく観測から機械的に導出する**。これは Motiva の設計において最も重要な原則の一つだ。

現在地の内部表現は**状態ベクトル（state vector）**である。状態ベクトルとは、現在の状況を複数の数値で表したもの（進捗率、品質スコア、未解決課題数、リソース消費率など）。各要素は単一の値ではなく「観測値 + 確信度」のペアで保持される。観測に実証根拠（テスト結果、ファイルの存在確認）がある場合は確信度が高く、ヒューリスティックな推定にとどまる場合は低い。

| 観測の種類 | 確信度 | 例 |
|------------|--------|-----|
| ファイルの存在確認 | 高 | 期待するファイルが生成された |
| テスト実行結果 | 高 | CI が通過した、テストが全件 pass |
| LLM による判断 | 低〜中 | 「これは完了しているか？」をアダプター経由で問い合わせ |
| ヒューリスティック | 低 | Write ツール呼び出し回数から進捗を推定 |
| 自己推定 | 低 | エージェント自身の「だいたい 60% 完了」発言 |

この設計により「なぜその行動を選んだか」がログ上で透明になる。

### 要素 3: ギャップ

ギャップはゴール（閾値）と現在地（状態ベクトル）の差分として機械的に計算される。ギャップの大きさが「今どれだけ動く必要があるか」を決め、次に生成するタスクの方向と優先度を駆動する。

ギャップの計算は確信度で重み付けされる。観測根拠が薄い次元のギャップは、実際より大きく見積もられる（確信度が低い = まだ確認できていない = ギャップが残っている可能性が高い）。これにより、未確認の項目を安易に「完了」と見なすことを防ぐ。

### 要素 4: 制約（3軸）

制約は生成されたタスクをフィルタリングし、エージェントの行動範囲を現実的に絞り込む。

**発散防止（量の制御）**: タスクの世代数・総数・ゴールからの意味的距離に上限を設ける。これにより AutoGPT のような再帰的発散を防ぐ。

**不確実性制約（質の制御）**: 確信度が低い状態では行動範囲を狭め、人間への確認を要求する。「よくわからないまま前に進む」を抑制する。

**リソース制約**: コンテキストウィンドウの残量・セッション時間・API 呼び出し残量に応じてタスクの重さを調整する。リソースが逼迫するほど、軽量なタスク（調査・確認）を優先し、重量タスク（大規模リファクタリング）を後回しにする。

## 2. オーケストレーションアーキテクチャ

Motiva は middleware でも hooks でもない。**Motiva がオーケストレーターであり、エージェントを呼ぶ側だ。**

```
ユーザー
  │ ゴール設定
  ↓
Motiva（永続プロセス / ループ）
  ├─ 動機エンジン（4要素モデル、スコアリング）
  ├─ 状態管理（ファイル永続化）
  ├─ セッション管理（開始 / 監視 / 終了）
  └─ アダプター層
       ├─ claude-code adapter  （claude CLI を呼ぶ）
       ├─ claude-api adapter   （Anthropic API を直接呼ぶ）
       └─ custom adapter       （任意のエージェント）
```

### ループ設計 — NEVER STOP

autoresearch（Andrej Karpathy の研究エージェント設計）と同思想:
ユーザーが止めるまで回し続ける。各イテレーションは独立したエージェントセッションだ。

```
Motiva ループ
    │
    ├─ ゴールから最優先タスクを選択（動機スコアリング）
    ├─ エージェントセッションを起動（タスクスコープを渡す）
    ├─ セッション完了を監視
    ├─ 結果を観測（ファイルシステム、テスト結果等）
    ├─ 状態ベクトルを更新
    ├─ 次のタスクへ or 完了判断
    └─ ループ継続
```

失敗時は keep/discard を判断して次へ進む。止まるべき理由が明示されない限り、ループは継続する。

### アダプターパターン — エージェント非依存

Motiva の実行層は交換可能だ。動機エンジン・状態管理・ループ制御はアダプターに依存しない。

| アダプター | 実行方式 |
|-----------|---------|
| claude-code | `claude` CLI コマンドをサブプロセスとして起動 |
| claude-api | Anthropic API を直接呼び、会話を管理 |
| custom | 将来: OpenClaw、Devin、任意のエージェント |

この設計により、動機構造の本体（4要素モデル、スコアリング、停滞検知）はエージェントの実装と切り離される。

## 3. 動機の3類型（動的スコア）

ゴールの優先度は固定テーブルではなく、時間と文脈で変動するスコアで管理する。タスクの選択はスコアが最も高い動機から行われる。

| 類型 | 動く理由 | タスク変換方式 | スコアの特性 |
|------|----------|----------------|--------------|
| 締切駆動 | 「いつまでにこの状態でなければならない」 | 逆算型 — 締切から逆算してマイルストーンを生成 | 締切が遠い間は低く、近づくにつれ急上昇 |
| 不満駆動 | 「今の状態がゴールに照らして不十分だから直す」 | 差分検出型 — ギャップが大きい次元を優先攻撃 | 放置・慣れで徐々に減衰 |
| 機会駆動 | 「今やると効率が良い、後で状況が変わる」 | イベント駆動型 — 好機の到来を検出して即実行 | 鮮度で急速に減衰（例: 12時間で失効） |

```
スコア
 ↑
 │    ━━━ 締切駆動（急上昇）
 │  ╌╌╌   不満駆動（緩やかに減衰）
 │ ·····  機会駆動（急速に減衰）
 │
 └────────────────────────────→ 時間
```

この動的スコアにより、「同じゴールでも今週と来週では優先度が変わる」という人間的な判断をモデル化できる。スコアは状態ベクトルの各要素（進捗率・品質スコアなど）から算出されるため、スコア変動の根拠も観測に基づいて透明になる。

## 4. メタ動機: 好奇心

好奇心は 3 類型の動機の**上位に位置するメタ動機**だ。3 類型が「既存のゴールをこなす」層であるのに対し、好奇心は**ゴール自体を生成・更新する**。

```
好奇心 ────→ ゴール生成・更新
  ↑
学習フィードバック（完了結果の蓄積）
```

### 発動条件

- タスクキューが空いたとき（「次に何をすべきか」が不明な状態）
- 想定外の結果を観測したとき（状態ベクトルが予測と大きく乖離した）
- 過去に失敗が多かった領域が存在するとき

### 学習フィードバックによる方向づけ

好奇心はランダムではなく、過去の学習によって方向づけられる。Motiva は「状態 → 行動 → 結果」のログを蓄積し、どの状態でどの行動が成功・失敗したかのパターンを分析する。

- 過去に状態ベクトルが大きく改善した領域（効果の高かった行動が存在する領域）を優先的に探索する
- 繰り返し失敗した領域を特定し、アプローチの再検討を提案する
- 成功パターンを別のゴールに転用できないか検討する

この学習によって、好奇心は時間とともに根拠ある方向に絞り込まれていく。

## 5. ゴール種別と階層

### ゴールの2種類: 起源による分類

Motiva のゴールはその起源によって 2 種類に分かれる。

**ユーザーゴール（origin: user）**

- ユーザーが明示的に設定したゴール
- 必達 — 自動的に却下・消去されない
- 常に好奇心ゴールより高い優先度を持つ
- 削除・クローズはユーザーのみが行える

**好奇心ゴール（origin: curiosity）**

- 好奇心エンジンが自動生成したゴール
- あくまで「提案」— ユーザーが拒否できる
- ユーザーゴールがすべて完了または停滞している場合にのみ追求する
- システムが自動で失効させられる（例: 12 時間のタイムアウト）

```
優先度
  高
  │  ユーザーゴール ───── 必達、人間のみが削除可能
  │
  │  好奇心ゴール ─────── 提案、自動失効あり
  低
```

### ゴール階層: タスク分解

ユーザーゴールは多くの場合、曖昧かつ大きい。これを直接追うのは不可能なので、子タスク・サブゴールに分解する必要がある。

**Motiva は LLM を直接呼んでゴールを分解する。**

Motiva はオーケストレーターとして、アダプター経由で LLM にゴール分解を依頼し、その結果を子ゴールとして内部に格納する。ホストエージェントへの「お願い」を経由する必要はない。

```
親ゴール（曖昧・大）
    │
    ├─ Motiva が LLM を呼んでゴールを分解
    │
    ├─ 子ゴール A（独自の達成閾値・観測ルール付き）
    ├─ 子ゴール B
    └─ 子ゴール C
        │
        ├─ Motiva が子ゴールごとにエージェントセッションを起動
        └─ 完了したら次の子ゴールへ（コンテキストリセット）
```

子ゴールの境界がそのままセッション境界になる。各セッションはスコープが明確で、コンテキストが汚染されない。

## 6. 進捗観測: 「カウントしない、観測する」

PoC での最大の失敗は、**進捗をツール呼び出し回数から盲目的に算出していたこと**だ。Write/Edit を 20 回呼ぶだけで進捗が 100% に達してしまう設計は、フィードバックループを持たない欠陥構造だった。

### 根本原則: 進捗は証拠から観測する

「何かを書いた」という行為ではなく「何が達成された」という証拠を観測する。

```
❌ 悪い進捗観測
   Write ツール呼び出し → progress + 0.05
   20回呼び出し → progress = 1.0 → ゴール完了

✅ 良い進捗観測
   期待ファイルが存在する → feature_completeness 確信度: 高
   テストが全 pass → test_coverage 確信度: 高
   README が更新された → docs_coverage 確信度: 中
   LLM に完了を確認した → 補助観測 確信度: 低〜中
```

### LLM による観測（補助的手段）

Motiva はアダプター経由で LLM に「このセッションの結果は完了しているか？」を問い合わせることができる。ただし LLM の判断は主観的な推論であるため、確信度は低〜中に設定し、ファイルシステム・テスト結果などの実証観測を上書きしない。LLM 観測は「実証根拠が薄い次元の補完」として機能する。

### 観測ルールの具体例

各ゴール次元には、その次元の達成を確認するための観測ルールを紐付ける。

| 次元 | 観測ルール例 | 確信度 |
|------|-------------|--------|
| feature_completeness | 定義した機能ファイルが存在し、0 bytes ではない | 高 |
| test_coverage | テストランナーが 0 エラーで終了した | 高 |
| build_success | ビルドコマンドが exit 0 で終了した | 高 |
| docs_coverage | README に必須セクションが存在する | 中 |
| completion（LLM） | LLM が「完了」と判断した | 低〜中 |
| progress（推定） | Write/Edit 呼び出し回数 | 低 |

### 進捗の上限規則

証拠に基づかない観測では、進捗を一定以上に上げられない。

```
証拠なし（ヒューリスティックのみ）: progress 上限 = 70%
テスト通過の証拠あり: 上限 = 90%
全次元に証拠あり: 上限 = 100%
```

これにより「何かやった気がする」だけで完了を宣言することを構造的に防ぐ。

## 7. 十分の判断（satisficing）

人間は完璧を目指さず、「これで十分」と判断して前に進む。エージェントは放置すると際限なく改善し続ける傾向がある。Motiva はゴールに達成閾値を持たせることで、この「十分の判断」をモデル化する。

### 完了判断のロジック

```
ギャップ < 閾値 ─→ 完了候補
        ↓
   確信度チェック
        ├─ 確信度 高 ─→ 完了
        └─ 確信度 低 ─→ 検証タスクを生成して確認してから完了
```

確信度が低い状態でギャップが閾値を下回ったとしても、それは「観測が不十分」を意味する可能性がある。そのため、完了宣言の前に検証タスク（例: 「テストを実行して結果を確認する」）を自動生成し、実際の達成を確認してから完了とする。

### なぜ閾値設計が重要か

閾値が 1 つしかない（例: `progress: 0.9` のみ）と、文章の書き込み回数だけでゴールが完了してしまう。多次元の閾値（`feature_completeness`, `test_coverage`, `docs_coverage` など）を設定することで、達成の意味が具体化される。これはゴール設定時にユーザーが明示するか、ゴール記述から自動導出することで実現する。

## 8. 停滞検知と切り替え

進捗が止まった状態を放置することは、エージェントにとっても人間にとっても非効率だ。Motiva は停滞を検知し、**自律的に対応策を実行する**。

### 検知指標

| 指標 | 閾値例 |
|------|--------|
| 見積もり時間の超過 | 予想時間の 2 倍を経過 |
| 連続失敗 | 同種アクションを 3 回連続で失敗 |
| 進捗変化率の低下 | 直近 N 回のイテレーションで状態ベクトルに変化なし |

### エスカレーション分岐

Motiva はオーケストレーターとして停滞に直接対処できる。「停滞を検知した」とエージェントに伝えるのではなく、Motiva 自身がセッション制御を行う。

```
停滞を検知
    │
    ├─ 情報不足 ──────→ 調査タスクを自己生成 → 新セッションで実行
    │
    ├─ 権限不足 ──────→ 人間にエスカレーション
    │
    ├─ 能力不足 ──────→ ゴールの再定義を人間に要請
    │
    └─ 外部依存待ち ──→ 現セッションを終了 → 別ゴールに切り替え
                        外部依存が解消されたら再開
```

現在のセッションを kill して別アプローチで新セッションを起動する、あるいはゴールをまるごと切り替えるといった判断を、Motiva が主導できる。

## 9. 人間との協調（信頼残高 × 確信度）

完全自律はゴールではない。Motiva は行動の自律度を 2 つの軸で**連続的に**調整する。これが既存エージェントとの本質的な差別化点だ（既存エージェントは「常に自信満々で行動する」か「常に人間に確認する」かの二択に陥りがち）。

Motiva はユーザーとエージェントの間に位置し、**全コミュニケーションを仲介する**。ユーザーの承認が必要かどうか、エージェントに何を伝えるかを Motiva が判断する。

### 2 つの軸

**信頼残高（trust balance）— 外部制御**

人間がエージェントにどれだけ任せるかを表す数値。成功のたびに少しずつ回復し、失敗のたびに大きく減少する（非対称設計）。失敗のペナルティを成功の報酬より大きくすることで、過信を防ぐ。ゴール単位で付与される。

**確信度（confidence）— 内部制御**

エージェント自身がどれだけ確かかを表す数値。観測根拠があれば高く、自己推定にとどまれば低い。これはエージェントが正直に「自分の知識の質」を評価するための仕組みだ。

### 4 象限の振る舞いマトリクス

```
            ┃   確信度 低        ┃   確信度 高
━━━━━━━━━━━╋━━━━━━━━━━━━━━━━━━╋━━━━━━━━━━━━━━━━━━
信頼残高 高 ┃ タスク生成するが   ┃ 自律実行
            ┃ 人間に確認         ┃
━━━━━━━━━━━╋━━━━━━━━━━━━━━━━━━╋━━━━━━━━━━━━━━━━━━
信頼残高 低 ┃ 現在地の確認       ┃ タスク生成するが
            ┃ タスクを先に生成   ┃ 人間に確認
```

### 不可逆アクションの特別扱い

信頼残高・確信度がどのレベルであっても、不可逆アクション（git push、本番デプロイ、テーブル削除、外部 API への書き込みなど）は**常に人間の承認を要求する**。

これは安全のフロアであり、いかなるスコアも上書きできない。

## 10. セッション継続性

Motiva は**セッションの境界を制御する側**だ。Motiva がセッションのライフサイクルを制御するため、コンテキストリセットのタイミングも Motiva が決定できる。

### セッション制御の設計思想

状態はすべてファイルに永続化される。各セッションはステートレスな実行単位であり、セッションが終わっても Motiva の状態は失われない。「中断」ではなく「保存と再開」だ。

```
Motiva ループ
    │
    ├─ ゴールから最優先タスクを選択
    │
    ├─ エージェントセッションを起動
    │       タスクスコープ・達成条件・制約を渡す
    │
    ├─ セッション完了を監視
    │       完了条件: 結果観測 / タイムアウト / 停滞検知
    │
    ├─ 結果を観測（ファイルシステム、テスト結果、LLM 判断）
    │
    ├─ 状態ベクトルを更新・永続化
    │
    ├─ 次のタスク選択 or 完了判断
    │
    └─ ループ継続（NEVER STOP）
```

### コンテキスト管理が解決される理由

コンテキストウィンドウの枯渇は、エージェントが「/clear してくれるだろう」という祈りに依存していたために問題だった。Motiva がオーケストレーターであれば、子ゴールの完了が自然なセッション境界になる。各セッションは新しいコンテキストで起動し、必要な情報だけを渡される。

- ユーザーは「前のセッションで何をしていたか」を改めて説明する必要がない
- Motiva が必要なコンテキスト（ゴール・制約・前回の結果）を組み立ててセッションに渡す
- セッションを跨いだ状態の一貫性は Motiva の永続ファイルが保証する

## 11. 学習フィードバック

Motiva は各セッションの経験を蓄積し、将来の判断を改善する。

### ログの構造

すべてのアクションは「状態 → 行動 → 結果」の形式でログに記録される。

- **実行前の状態**: 状態ベクトルのスナップショット
- **実行したアクション**: 何をしたか
- **実行後の状態**: 状態ベクトルの変化
- **結果**: 成功 / 失敗 / 停滞

### フィードバックの方向

| 用途 | 内容 |
|------|------|
| 好奇心への入力 | 状態ベクトルが大きく変化した領域を「効果的な探索対象」として推薦 |
| タスク生成への入力 | 過去に成功したアプローチを同種タスクで優先 |
| 停滞検知への入力 | 過去に行き詰まったパターンを早期に認識し、早めにエスカレーション |
| 信頼残高の校正 | 実績ベースの信頼残高更新（成功/失敗の記録） |

学習により、Motiva は時間とともに「このプロジェクトでは何が効く」を内在化していく。これは単なるタスクキューではなく、経験から学ぶ動機エンジンとしての Motiva の本質的な価値だ。

## まとめ: Motiva がどう「動かす」か

以上の 11 のメカニズムをつなぐと、Motiva の動作原理がひとつの流れとして見えてくる。

```
ユーザーがゴールを設定
    │
    ├─ Motiva が LLM を呼んでゴールを分解（子ゴール群へ）
    └─ 動的スコアを計算（締切/不満/機会）
    │
Motiva ループ（NEVER STOP）
    │
    ├─ 最優先タスクを選択
    ├─ エージェントセッションを起動（スコープ・制約を渡す）
    ├─ セッション完了を監視
    │
    ├─ 証拠ベースで状態ベクトルを更新
    ├─ ギャップを再計算
    ├─ satisficing: 完了判断 or 検証タスク生成
    ├─ 停滞: 検知 → セッション kill or ゴール切り替え
    └─ 不可逆: 人間承認を強制
    │
セッション終了ごとに
    │
    ├─ 状態を永続化
    ├─ ログ追記 → パターン分析
    ├─ 信頼残高更新
    └─ 好奇心エンジンへのフィードバック
```

Motiva はエージェントではない。エージェントを動かすオーケストレーターだ。知能は既にある。足りないのは「何のために動くか」「いつ止まるか」「次に何をするか」を永続的に管理する動機構造だった。
